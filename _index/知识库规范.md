# 📐 知识库规范指南

> 本文档定义知识库的目录结构、命名规范、笔记标准和维护流程。
> 目标是让知识库随笔记量增长而**自然有序**，而不是越来越乱。

---

## 二、目录命名规范

| 元素 | 规范 | 示例 |
|------|------|------|
| **领域目录** | `序号-领域名` (中文, PascalCase 语义) | `01-开发`, `02-AI` |
| **子目录** | 中文简短名词 | `后端`, `前端`, `数据库` |
| **技术子目录** | 英文技术名（首字母大写） | `Java`, `Vue`, `MySQL` |
| **MOC 目录** | 固定 `_index` | `_index/MOC-开发.md` |
| **收件箱** | 固定 `inbox` | `inbox/` |

### 命名原则

1. **归类而不嵌套**：尽量不超过 4 层：`01-开发/前端/Vue/组件`
2. **目录即标签**：目录名就是笔记的隐式标签
3. **序号排序**：数字越小越重要/使用频率越高
4. **一致性**：已存在的英文名沿用，不中英混用

## 三、笔记命名规范

### 3.1 文件名

| 规则 | 说明 |
|------|------|
| **中文笔记** | 使用有语义的中文名称 |
| **英文技术名** | 保留英文 (如 `maven生命周期1`) |
| **禁止无意义命名** | ❌ `未命名.md`, ❌ `temp123.md`, ❌ `新建文档.md` |
| **数字后缀** | 仅用于系列笔记，且应聚合为 MOC |

### 3.2 Frontmatter（YAML 头部）

每篇笔记 **必须** 包含以下 frontmatter：

```yaml
---
created: 2026-06-20          # 创建日期
updated: 2026-06-20          # 最后更新（按需）
tags:                        # 至少一个标签
  - java/maven
  - 基础
aliases: [别名1, 别名2]      # 可选，方便查找
type: note                   # note | moc | project | guide
---
```

### 3.3 Frontmatter 字段说明

| 字段 | 必填 | 说明 |
|------|:----:|------|
| `created` | ✅ | 创建日期，格式 `YYYY-MM-DD` |
| `updated` | 🟡 | 有修改时更新 |
| `tags` | ✅ | 至少 1 个，使用 `/` 层级 |
| `aliases` | 🟡 | 常用搜索名 |
| `type` | 🟡 | 笔记类型：`note` / `moc` / `project` / `guide` |

## 四、笔记内容规范

### 4.1 笔记类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **概念笔记** | 解释一个技术概念 | `maven基础概念.md` |
| **教程笔记** | 步骤式操作指南 | `vue核心.md` |
| **问题排查** | 记录 bug 及解决方案 | `问题修复1.md` |
| **速查表** | 常用命令/配置汇总 | `命令.md` |
| **MOC** | 领域索引 Map of Content | `知识图谱.md` |
| **剪藏** | 外部文章存档 | Clippings 目录下的文件 |

### 4.2 内容结构建议

```markdown
# 标题（1 级标题）

## 一句话总结
> 如果只能记住一句话，应该是什么？

## 核心概念
...

## 展开说明
...

## 相关笔记
- [[相关笔记1]]
- [[相关笔记2]]

## 参考来源
- [外部链接](url)
```

### 4.3 链接规范

| 场景 | 语法 | 说明 |
|------|------|------|
| 链接库内笔记 | `[[笔记名]]` | **必须使用** |
| 链接到标题 | `[[笔记名#标题]]` | 精确引用 |
| 嵌入图片 | `![[图片.png]]` | 图片放 `assets/` |
| 外部链接 | `[文字](url)` | 仅用于外部 |

> **黄金法则**：每篇新笔记至少与 1 篇已有笔记建立 `[[wikilink]]`。孤儿笔记 = 遗忘笔记。

## 五、收件箱（inbox）制度

### 5.1 使用流程

```
随手记录
   ↓
inbox/ 暂存        ← 快速记录，无需分类
   ↓
每周回顾
   ↓
三叉戟决策 ──→ 归档到领域目录
         ├──→ 合并到已有笔记
         └──→ 删除（无用内容）
```

### 5.2 规则

- **新笔记一律先放 `inbox/`**，不要直接创建到领域目录
- **每周至少清理一次 inbox**，保持 inbox 不超过 10 篇
- inbox 中的笔记不需要完美格式，内容优先

## 六、标签规范

### 6.1 标签层级

```
#领域/子领域/具体技术

示例：
#java/maven/生命周期
#vue/组件/通信
#前端/环境/配置
#工具/docker
#finance/会计
#ai/llm
#meta/index      ← 元信息（图谱、规范等）
#cheatsheet      ← 速查表
#wip             ← 未完成笔记
```

### 6.2 标签原则

1. 每个笔记最少 1 个标签，最多 5 个
2. 使用 `/` 表示层级，不要用 `:` 或 `-`
3. 不要创建孤立标签（至少 2 个笔记使用）
4. `#wip` 表示未完成，用于占位笔记

## 七、图片 / 附件管理

- **所有图片统一放入** `assets/`
- 粘贴截图后，在 Obsidian 设置中将附件路径设为 `assets/`
- 图片命名：`{描述}_{日期}_{序号}.png`
  - 如 `docker-network-20260620-01.png`
  - ❌ `Pasted image 20260315011324.png`（难以识别）
- 文档中的图片使用 `![[assets/图片名.png]]` 嵌入

---

## 八、Git 规范（如果使用版本管理）

- **有意义的一次提交**：描述变更范围，如 `feat: 新增 Docker 网络笔记`
- **避免提交**：`assets/` 中的大二进制文件可考虑 `.gitignore`
- **提交频率**：每周一次，与 inbox 清理同步

---

## 九、知识库健康检查清单

- [ ] inbox 笔记 < 10 篇
- [ ] 无孤儿笔记（零链接笔记）
- [ ] 知识图谱已更新
- [ ] 所有新笔记含 frontmatter
- [ ] 图片已重命名并移到 `assets/`
- [ ] 无空 `未命名.md` 文件

---

> **最后原则**：规范是工具，不是枷锁。当规范阻碍记录时，优先记录，再回头整理。
> 一个好的规范应该让「随手记」和「有序」不再矛盾。
