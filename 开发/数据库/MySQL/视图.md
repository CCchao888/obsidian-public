
## 前端一个表对应后台多个底表

---

### 方法一：使用 `UNION ALL`（最推荐，代码控制力强）

**核心思想：** 在后端写 SQL 语句时，使用 `UNION ALL` 关键字，把两张表的数据临时“摞”在一起，当成一张大表来查。

**⚠️ 关键前提：** 两张表 `SELECT` 出来的列数、顺序和数据类型必须完全一致。

**具体步骤：**

你的后端在处理这个分页请求时，需要执行**两次**数据库查询。

**1. 查数据总条数 (Total Count)**

Webix 的分页器必须知道总共有多少页，所以你需要查出两张表的总行数。

SQL

```
-- 假设两张表叫 table_a 和 table_b
SELECT COUNT(*) AS total_count 
FROM (
    SELECT id FROM table_a
    UNION ALL
    SELECT id FROM table_b
) AS combined_table;
```

_(注：这里只 SELECT `id` 是为了提高统计速度，不需要把所有列都查出来。)_

**2. 查当前页的数据 (Paged Data)**

把两张表合并后，统一进行排序和分页（以 MySQL 的 `LIMIT` 为例，假设取第一页，每页 20 条）。

SQL

```
SELECT * FROM (
    -- 明确写出需要展示的列，确保两张表顺序一致
    SELECT id, user_name, status, create_time FROM table_a
    UNION ALL
    SELECT id, user_name, status, create_time FROM table_b
) AS combined_table
ORDER BY create_time DESC  -- 按照时间倒序排列
LIMIT 0, 20;               -- 分页：从第 0 条开始，往后取 20 条
```

> **💡 为什么要用 `UNION ALL` 而不是 `UNION`？**
> 
> `UNION` 会在合并后自动执行一次“去重”操作，这会极大地消耗数据库性能。既然是两张不同的底表，通常数据不会重复，所以直接用 `UNION ALL`（只拼接，不去重）速度最快。

---

### 方法二：创建数据库视图 View（最优雅，后端代码最干净）

**核心思想：** 直接在数据库里建一个“虚拟表”（视图）。这个视图内部包含了拼接的逻辑，但对外暴露的样子和普通表一模一样。

**具体步骤：**

**1. 在数据库中创建视图（只需执行一次）**

你可以让你们的 DBA（数据库管理员）或者你自己，在数据库客户端里运行这段 SQL：

SQL

```
CREATE VIEW view_combined_data AS
SELECT id, user_name, status, create_time FROM table_a
UNION ALL
SELECT id, user_name, status, create_time FROM table_b;
```

**2. 后端代码把它当成单表查询**

创建好视图后，你的后端代码就彻底解脱了！后端不需要知道底下有两张表，直接把 `view_combined_data` 当成一张普通的单表来写分页逻辑。

查总数：

SQL

```
SELECT COUNT(*) FROM view_combined_data;
```

查分页：

SQL

```
SELECT * FROM view_combined_data ORDER BY create_time DESC LIMIT 0, 20;
```

---

### 总结与对比

|**特性**|**方法一：UNION ALL (写在后端)**|**方法二：创建视图 View (建在数据库)**|
|---|---|---|
|**优点**|灵活，修改查询条件不用动数据库结构；开发人员自己就能搞定。|后端代码极度简洁，逻辑解耦；如果是复杂的拼接，视图可以重复被不同的接口调用。|
|**缺点**|SQL 语句会比较长，如果表结构改了，代码也要跟着改。|需要在数据库里创建对象，有的公司对开发人员没有开放这种权限。|

不管你用哪种方法，最后你的后端接口只需要给 Webix 前端返回类似这样的 JSON 格式即可：

JSON

```
{
  "total_count": 150, 
  "data": [
      // 这里是 LIMIT 返回的 20 条数据
  ]
}
```
